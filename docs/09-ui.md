# 09 · La UI: dashboard de revisión y aprobación

Un sistema editorial necesita **un humano en el loop** antes de publicar. La UI es
un dashboard **FastAPI + HTMX** (server-side, sin build de frontend) para revisar
cada editorial, editarlo y aprobarlo.

## Levantarla

```bash
make web          # o: python -m editorial_team.webapp.app
# abre http://localhost:8080
```

## Qué hace

| Pantalla | Acción |
|----------|--------|
| **Inicio** (`/`) | Lista el historial de corridas con su estado (`generated`/`published`/`failed`). Botón **“Generar editorial”** lanza una corrida en segundo plano. |
| **Detalle** (`/runs/{id}`) | Muestra el editorial renderizado + imagen, el **fact-check**, la **revalidación crítica** y las **variantes de redes**. |
| Editar | Form para corregir título, bajada y cuerpo antes de publicar. |
| **Aprobar y publicar** | Publica de verdad en blog + redes (ignora `DRY_RUN`), vía HTMX (actualiza el estado sin recargar). |

## Cómo encaja con el resto

```
run_daily / Scheduler ─► run_pipeline() ─► SQLite (storage.py)
                                              ▲
        Dashboard (webapp) ───── lee/edita ───┘
                          └── "aprobar y publicar" ─► publish_run() ─► WordPress + upload-post
```

- El pipeline **persiste** cada corrida en SQLite (`storage.py`).
- La UI lee esas corridas y permite editar el borrador (`services.save_draft_edits`).
- “Aprobar y publicar” llama a `services.publish_run`, que reusa la misma lógica de
  publicación que el nodo `publisher` (`publishing.do_publish`) pero forzando
  `dry_run=False`.

## Endpoints

| Método | Ruta | Para qué |
|--------|------|----------|
| GET | `/` | panel |
| GET | `/runs/{id}` | detalle |
| POST | `/runs/{id}/edit` | guardar ediciones |
| POST | `/runs/{id}/publish` | publicar (blog + redes) |
| POST | `/run` | lanzar una corrida (background) |
| POST | `/tasks/run-daily` | lo dispara Cloud Scheduler (protegido por `SCHEDULER_TOKEN`) |
| GET | `/healthz` | health check (Cloud Run) |

## Flujo recomendado (human-in-the-loop)

1. El Scheduler corre el pipeline con `DRY_RUN=true` → genera y **persiste** el editorial.
2. Entrás al dashboard, revisás datos/crítica, editás si hace falta.
3. Tocás **“Aprobar y publicar”** → sale en blog + redes.

> ⚠️ El dashboard no trae autenticación propia. En local está OK; en producción
> protegelo (IAM/IAP en Cloud Run) — ver [10-deploy-gcp.md](10-deploy-gcp.md).
